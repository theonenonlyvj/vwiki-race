# QF-10 [P1/S] Native share sheet on mobile

## Files
- src/race/shared.tsx:49-88, 136-180

## Changes
Feature-detect navigator.share inside the share flow: on supporting devices (iOS Safari, Android Chrome) one tap opens the OS share sheet with the already-composed spoiler-safe text; the existing clipboard + 'Paste it anywhere' toast remains the untouched fallback for desktop/unsupported browsers. Treat user-cancelled share (AbortError) as a no-op, not an error.

## Acceptance
- Tapping Share result on a phone opens the native sheet landing directly in Messages/WhatsApp
- Desktop behavior byte-identical to today; cancelling the sheet shows no error state
- Share text content unchanged (invariant 1: always time + clicks, spoiler-free)

## Risk
Low — additive feature-detected branch over an already-tested fallback.

## Judge A (amend)
1. Widen scope to ChallengeShareButton (race/shared.tsx:182-215), or explicitly justify excluding it. It's the same file, built on the same useClipboardShare hook, and shares the exact clipboard-only problem (used for "Copy challenge link" invites — arguably an even more natural native-share use case than bragging about a score). Shipping native share only on ShareResultButton leaves two adjacent, near-identical affordances in one component behaviorally inconsistent right after a round explicitly about brand/button consistency — that's the kind of gap this council flags next time. Cheapest fix: generalize the hook (e.g. `useShare(text)` wrapping both navigator.share and the existing clipboard fallback) so both buttons pick it up for the same S effort.

2. Fix acceptance-criteria wording: "opens the native sheet landing directly in Messages/WhatsApp" overstates navigator.share's behavior — it opens the OS's general share picker where the user chooses a target; it doesn't deep-link into a specific app. Reword to "opens the native OS share sheet, with Messages/WhatsApp among the one-tap targets" so nobody implements/tests against a false expectation.

3. Implementation note to carry into the diff: composeShareText (shared.tsx:116-128) already bakes the URL into one combined string. The share() call should pass that whole string as `text` only (no separate `url` field) — passing both would risk some share targets (notably Messages on iOS) displaying the link twice. Worth a one-line acceptance item so this isn't left as an implementer's guess.

### Judge A reasoning
Verified against code and evidence. Read src/race/shared.tsx in full: useClipboardShare (49-88) and ShareResultButton (136-180) are cited correctly and are exactly what the package describes — a clipboard-write-with-execCommand-fallback flow with no navigator.share anywhere. Grepped the whole src tree for "navigator.share"/"navigator.canShare" and found zero hits (only unrelated AbortError usage in network-abort tests), confirming the gap is real, not already partially addressed. Confirmed ShareResultButton/composeShareText are consumed by both Home.tsx (post-play card) and RaceResults.tsx (Results), so the fix's reach matches the "reused component" framing PKG-05 itself established.

Checked round-1 record for prior rulings on this surface: PKG-05 (Results share-first rework, shipped) touched this exact component but only for hierarchy/DNF-gating/routing/copy — never the transport mechanism (clipboard vs. native share). So this is not a re-litigation of a shipped decision; it's a genuinely new, previously-undiscussed capability, which is exactly the "find what's next" brief for this round.

On owner-fit: additive, feature-detected, explicitly preserves desktop behavior byte-for-byte and treats cancel as a no-op — low risk matches the stated S/Low. It targets the one viewport the app is designed around (390px) and the one action that turns a solo daily habit into a shared one, which matters more, not less, given the ~5-player real community stated as a hard constraint — reducing friction on invite/brag is pro-growth, not scope creep, for a Wordle-style ritual game. It doesn't invent new visual language, doesn't touch the in-flight HUD-timer hotfix or the daily-drop state also called out as out-of-scope, and reuses the already-tested share-text composition (invariant 1) rather than touching it.

The three amendments are real but non-blocking: (1) an inconsistency the package leaves on the table right next to the code it's already touching, (2) a factual overstatement in the acceptance criteria about how navigator.share's UI behaves, (3) an implementation ambiguity (text-only vs. text+url) that's cheap to specify now and easy to get wrong silently later. None of these are reasons to drop; all are cheap to fold into the same S-effort diff. Keep as-is would risk shipping the ChallengeShareButton inconsistency and the acceptance-criteria wording mismatch without anyone flagging them — hence amend, not keep outright.

## Judge B (amend)
1) Name test requirements explicitly: add cases for navigator.share present+resolves (asserts called with the same composed text; clipboard NOT written), present+AbortError (treated as no-op, no failed state shown), and absent (existing App.test.tsx clipboard tests, e.g. lines 3835-3963, must keep passing unmodified since jsdom has no navigator.share). 2) Resolve the ambiguous file scope: the change must live locally inside ShareResultButton's onClick (shared.tsx:136-180) only — do NOT modify the shared useClipboardShare hook (49-88), since it's also consumed by ChallengeShareButton ("Copy challenge link" in ChallengeDetail.tsx:196) which is out of this package's acceptance criteria and must not silently start opening the OS share sheet too. 3) Correct the device assumption: desktop Safari (macOS) and desktop Chrome/Edge (Windows) also implement navigator.share, so a bare `if (navigator.share)` check will change desktop behavior on those browsers, contradicting the 'Desktop behavior byte-identical to today' acceptance line, and nothing in CI (jsdom lacks navigator.share) will catch the divergence. Either explicitly accept broadened desktop native-share and rewrite that acceptance criterion, or add a coarse-pointer/mobile guard if strict desktop parity is actually required. 4) Add a pending/in-flight UI state: disable the Share result button while navigator.share()'s promise is outstanding (mirroring the existing disabled={copyStatus === 'copying'} pattern) to prevent a double-tap from firing two overlapping share sheets.

### Judge B reasoning
Verified against the live file (src/race/shared.tsx): line citations for useClipboardShare (49-88) and ShareResultButton (136-180) are accurate, and 'invariant 1' is a real term in the design spec. The fallback path is heavily pre-tested (8+ tests in App.test.tsx directly click 'Share result' and assert clipboard writeText calls), which confirms the plan's 'already-tested fallback' / 'byte-identical' claim will hold in CI since jsdom doesn't implement navigator.share. The idea itself is additive, low-risk, doesn't undo any round-1 shipped package, and correctly preserves invariant 1. However the plan as literally written has real implementability gaps: no named test requirements (violates this cycle's judge mandate), an ambiguous edit scope that risks bleeding into the shared useClipboardShare hook and thus ChallengeShareButton's untested 'Copy challenge link' behavior, a factually incomplete device assumption (desktop Safari/Chrome also support navigator.share, threatening the stated desktop-parity acceptance criterion with no CI safety net), and no mention of a pending-state guard against double-invocation. These are all spec-able in the brief itself without owner/design input, so amend rather than drop or keep as-is.
