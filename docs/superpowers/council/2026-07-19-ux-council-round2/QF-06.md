## Changes
When today's daily hasn't dropped, the Today pill's card gets honest framing ('Nothing new yet — here's yesterday's board') instead of silently rendering a byte-identical duplicate of the Yesterday tab headed 'YESTERDAY'S DAILY'. Give useDailyCountdown a one-shot onExpire callback that bumps catalogRefreshVersion (the exact trigger the focus listener already uses) so a foreground tab self-heals across the 5 AM boundary. Un-gate Results' daily-cadence line: render the 'New daily drops 5:00 AM Central' countdown line on every daily Results (completed AND DNF), keeping the '🔥 Day 1' framing as an additive first-finish variant — but with 'come back tomorrow for the next one' instead of 'defend your spot'. On Home, when dailyStreak > 0 and today is unplayed, merge streak + deadline into one line ('🔥 5-day streak — 2:14 left to keep it').

## Acceptance
- At the pre-drop boundary, Today and Yesterday are no longer visually identical and Today never contains a card calling itself yesterday's
- A tab left focused through 5:00 AM Central shows the new daily within a minute, no blur/focus needed
- Every daily Results screen (Day 1 or Day 100, finish or DNF) ends with a forward-looking drop-time cue
- Mid-streak unplayed Home leads with the combined streak-expiry line

## Risk
Low-medium — the rollover callback touches timing logic; mitigated by reusing the existing focus-refresh path rather than new fetch machinery.

### Judge A reasoning
Verified each of the five "covers" claims against code and the cited screenshots before ruling.

The package's lead item is a hard violation, not a taste call. Boards.tsx:96-115 carries an explicit code comment: "Owner-proxy ruling (2026-07-19 council): Today and Yesterday briefly rendering the identical board pre-drop is intentional, not a redundancy bug — pre-drop, the current daily genuinely IS yesterday's, so the two tabs agreeing is correct." I traced this to docs/superpowers/council/2026-07-19-ux-council/PKG-01.md line 65: "OWNER-PROXY RULING (binding): Amend-A choice = (a) — pre-drop, Boards' Today mirrors Home's honest yesterday-daily framing... Today and Yesterday briefly showing the same board is correct, not a bug." That ruling explicitly considered and rejected option (b) — "Have Today render its own honest empty state pre-drop" — which is word-for-word what QF-06 now proposes ("the Today pill's card gets honest framing ... instead of silently rendering a ... duplicate of the Yesterday tab"). This is the exact re-litigation the round-2 brief told the council not to do, wrapped inside an otherwise-legitimate-sounding retention package where it's easy to wave through unnoticed. The evidence framing also overstates the bug: mobile-03-boards-today.png vs mobile-03-boards-yesterday.png show real differences (Today carries the countdown line and the Race CTA, distinct badge phrasing) — only the placement/DNF rows match, because they're the same challenge, which is the ruled-upon, intended behavior. This sub-finding should not ship in any form without going back to the owner and explicitly overturning yesterday's binding call — that's a BACKLOG/owner conversation, not a weekend quickfix, and shouldn't ride along here.

The other four items hold up and are worth doing now:
- 5 AM self-heal: confirmed at App.tsx:377-395 — the catalog only refreshes on window `focus`/`visibilitychange`, no timer, so a tab kept foregrounded straight through the 5:00 AM Central boundary never re-fetches. The proposed fix (useDailyCountdown gets a one-shot onExpire that calls the same setCatalogRefreshVersion bump the focus listener already uses) is a real, low-risk reuse of existing plumbing, not new fetch machinery.
- Results never showing a forward-looking daily cue outside the one-time ritual: confirmed in RaceResults.tsx — the only "New daily drops" text anywhere on that screen lives inside `showFirstFinishRitual`, gated to `outcome.status === "completed" && preRaceCompletions === 0`. A DNF, or literally any run after your first-ever completion, closes with no cadence cue at all. This is a genuine, previously-unaddressed retention gap, independent of the Today/Yesterday ruling.
- "🔥 Day 1 · New daily drops 5:00 AM — come defend your spot" is literal code (RaceResults.tsx:261) and is objectively odd on a player's first-ever run (nothing exists yet to defend). Cheap, sensible copy fix.
- Streak+countdown never combined: confirmed — Home.tsx renders the countdown as a `ritual-line` inside the hero card (line 254-256) and the streak as a separate `StreakTrendRow` block further down (line 280, defined 396-419); they're two independently-shipped elements, not a single ruled-upon design, so merging them for a returning player is additive economy rather than a reversal of anything. Reasonable taste-positive polish, though it's a real (if small) layout consolidation across two components, not a one-line copy tweak — worth flagging so the M estimate doesn't get squeezed.

Net: three-fifths of this package's claims are solid, low-risk, genuinely-new-for-round-2 fixes that fit "worth doing now." One-fifth (Today-tab framing) must be struck because it directly contradicts a binding ruling from the round-1 council on the exact same screen, and its "byte-identical" framing overstates what the screenshots actually show. Amend and ship the rest.

### Judge B reasoning
Three of the five "covers" items are real, correctly-cited, unshipped gaps with reasonable low/medium-risk plans — but one sub-item re-litigates an already-shipped, tested, owner-ruled round-1 decision, and the implementable-as-written bar fails on two mechanisms.

VERIFIED AGAINST CODE:

1. Boards.tsx Today framing ("cold-critic: Today tab contradicts its label") is INVALID — it's already fixed and explicitly ratified. Boards.tsx:575-616 (PKG-01) already renders honest, distinct framing: pre-drop, Today shows a "Yesterday's daily · <flavor>" badge (renders visually as "YESTERDAY'S DAILY · HARD · DAILY #4" per screenshot mobile-03-boards-today.png) plus a "New daily drops 5:00 AM Central." line that the Yesterday tab (mobile-03-boards-yesterday.png) does NOT have — they are not byte-identical, and Today never carries an unqualified "Today" label. Boards.test.tsx:120-139 is a named test literally titled "kind yesterday-daily (pre-drop): Today mirrors Home's honest framing - no unqualified TODAY label" that asserts this exact behavior as correct. The code comment at Boards.tsx:228-233 records the owner-proxy ruling: "Today and Yesterday briefly showing the identical board is intentional... not a redundancy bug." Implementing this sub-item as written would directly contradict a round-1 ruling on file — exactly what the round-2 brief prohibits. Strike it (and drop it from "covers").

2. Self-heal mechanism is architecturally incomplete. useDailyCountdown is only ever called from Home.tsx:170, and AppShell.tsx:224 conditionally renders `visibleMode === "home" ? <Home/> : ...` — Home fully unmounts on Stats/Challenges/You. So an onExpire wired through Home's hook only fires if the user happens to be parked on the Home tab at 5 AM; a tab left on Boards' Today segment (very plausible — arguably more likely, since Home's whole job is to launch you into a race) will NOT self-heal, contradicting the acceptance bar as written ("a tab left focused... no blur/focus needed," unqualified). Move the boundary trigger to App.tsx itself (mode-independent): schedule a single setTimeout off the already-tested `msUntilNextCentralDrop(new Date())` (domain/dailyCountdown.ts) inside the existing effect at App.tsx:377-395, bumping catalogRefreshVersion directly and rescheduling after each reload — simpler than a per-second "did the value jump" heuristic bolted onto Home's hook, and removes the mount-dependency gap.

3. Wrong gating condition risk on RaceResults. RaceResults.tsx:239 computes `isDailyToday = isChallengeDailyToday(challenge, todayCentral)`, which is false for a challenge that IS a daily but not literally today's — exactly the pre-drop case (racing yesterday's still-playable daily) that this very council cycle is supposed to judge the presentation of. If the un-gated cadence line is naively keyed off the obviously-named `isDailyToday` already in scope at that call site, it would silently exclude the pre-drop state. Gate on daily-ness generically instead (`challenge.dailyFeature` truthy / `dailyDateForChallenge(challenge) !== null` — the same signal Boards.tsx/Home.tsx already use for their flavor badges).

4. Test requirements aren't named, though this repo's convention is dense, explicitly-named test coverage for exactly this kind of behavior (Boards.test.tsx's "PKG-01" describe block; App.test.tsx:3425-3498 locks the literal "come defend your spot" string the package intends to change; App.test.tsx:4186-4481 locks "left today" phrasing). Name at minimum: src/hooks/useDailyCountdown.test.ts (or a new App-level test) for the mode-independent boundary trigger, and App.test.tsx updates for the new RaceResults line (finish+DNF, day-1+non-day-1) and the merged Home line — and flag that the copy change will break the existing literal-string assertions at App.test.tsx:3440/3463/3484/3498 so the implementer isn't surprised by red tests.

5. Home.tsx citation (254-281) is too narrow for the streak+countdown merge — the actual composition logic lives in the shared `StreakTrendRow` function (Home.tsx:396-420, also reused unmodified at Home.tsx:280 finished-state... i.e. 357 and in "You"), so merging for the pre-play-only case needs either a new optional prop there or a parallel composition path that doesn't disturb its other call sites. Also, the illustrative merged copy "2:14 left to keep it" doesn't match what `formatCountdown` (domain/dailyCountdown.ts) actually emits (hardcoded "...left today" suffix) — either keep "left today" in the merged line (zero domain-layer risk) or add domain/dailyCountdown.ts + its test file to scope if a distinct suffix is wanted.

6. Minor consistency gap: Home.tsx already has an unlisted second instance of "come defend your spot" in its own post-play ritual line (Home.tsx:359-361, fires on every finish, not just Day 1) — the same social-dynamics issue the package fixes in RaceResults. Worth folding in for consistency or explicitly deferring with a reason.

The underlying problems in items 2-5 (once corrected) are real, P1-appropriate, and low/medium risk — this shouldn't be dropped outright, just corrected before build.
