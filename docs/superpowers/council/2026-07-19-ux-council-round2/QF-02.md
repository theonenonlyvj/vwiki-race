## Changes
Add '/assets/* Cache-Control: public, max-age=31536000, immutable' to _headers (index.html stays max-age=0). In sanitizeAttributes, unconditionally set loading="lazy" and decoding="async" on article <img> instead of only whitelisting pre-existing values. Lazy-load AdminDailies via React.lazy behind the existing isAdminDailiesRoute() gate. Add preconnect for en.wikipedia.org + the API worker host and preload the Luckiest Guy / Fredoka woff2s in index.html. Add a session-scoped Map cache in Boards for closed segments (yesterday/finished trend windows) only — never 'today'.

## Acceptance

- Sanitized article HTML: every img carries loading=lazy decoding=async
- AdminDailies is a separate chunk; main bundle shrinks; non-admin path unaffected
- Bouncing Today→Yesterday→Today issues no repeat network call for the closed board

## Risk

## Judge A (amend)
1. LEAD-IMAGE EXCLUSION (must-fix before merge): Don't blanket-apply loading="lazy" to every article <img>. Confirmed via flow-03-race-screen.png and flow-04-race-after-click.png that the lead image (Moon photo, Near-side-of-the-Moon diagram) renders at the very top of the viewport with zero scrolling on literally every article the player lands on — it's the LCP element of the core race loop, repeated dozens of times per race. Lazy-loading it risks a measurable perceived-speed regression on exactly the interaction this game is about ("how fast can I parse this page and find my link"), which cuts against the perf package's own goal. Fix: in sanitizeTree's document-order walk (wikipediaSanitizer.ts:355-374), keep the first <img> eager (optionally fetchpriority="high"); lazy-load the rest. Small, same-scope change — not new work.

2. BOARDS CACHE SCOPE: "finished trend windows" isn't a real state — 7d/30d/lifetime keep absorbing today's results until midnight. Session-caching them risks a player finishing their daily race, flipping to Boards, and not seeing their own fresh run — a gap a 5-player community will notice and mention same-day. Narrow the cache to "yesterday" only (the one segment that's actually and permanently frozen), or add explicit cache invalidation the moment the player submits a run this session.

3. FONT PRELOAD MECHANICS: don't hand-write `<link rel=preload href="/assets/fredoka-...-HASH.woff2">` in index.html. Confirmed live: dist/assets/fredoka-latin-400-normal-17JuUzdy.woff2 — Vite content-hashes this filename on every build, so a static tag goes stale on the very next deploy (either 404s or silently preloads nothing). Either drop the font-preload half of this line item (the preconnect half is safe, free, and fine as-is) or implement via a small transformIndexHtml Vite hook that reads the manifest.

Everything else checks out and should ship as-is:
- Verified LIVE (curl against vwikirace.pages.dev just now): /assets/index-BqFtIVtz.js currently returns "cache-control: public, max-age=0, must-revalidate" — the P0 claim is real, not speculative.
- The public/_headers mechanism is the correct, standard Cloudflare Pages pattern and matches the repo's existing public/_redirects → dist/_redirects precedent; index.html correctly keeps getting max-age=0 by Cloudflare's own default (confirmed via the same curl — no explicit rule needed for it).
- AdminDailies confirmed statically imported at AppShell.tsx:2 and bundled into the single 329KB dist/assets/index-*.js (no code-splitting anywhere in this codebase today) — lazy-loading it behind the existing isAdminDailiesRoute() gate is real, low-risk, and invisible to the ~5 real players.
- wikipediaSanitizer.ts:384-398 (sanitizeAttributes/sanitizeTokenAttribute) confirmed to only whitelist a value the source HTML already supplied and never adds loading/decoding when absent — today essentially zero article images get any lazy hint. The package's description of current behavior is accurate.
- Boards.tsx:235-252 confirmed the today/yesterday useEffect refetches on every activeChallenge?.id change with no memoization — bouncing segments costs a real repeat network call today.

### Judge A reasoning
Verified this package's factual claims directly rather than taking them on faith, and they hold up: the P0 caching bug is real and currently live (curled vwikirace.pages.dev and got cache-control: public, max-age=0, must-revalidate on a content-hashed JS asset), the _headers mechanism proposed matches an existing precedent in this exact repo (public/_redirects), the sanitizer and AdminDailies claims match the code exactly at the cited lines, and the Boards refetch claim matches the actual useEffect logic. This is a coherent, single-themed "network/caching" package (not scope creep — it's the same granularity as round-1 packages), it doesn't touch any shipped visual/IA decision, and it directly serves "worth returning to daily" via faster repeat loads for a tiny but real community.

However, cross-referencing the fix against the actual product (not just the code) surfaced a real defect in the plan: this is a Wikipedia-racing game where every single navigation lands on an article whose lead image sits at the very top of the viewport (confirmed in flow-03/flow-04 screenshots) — blanket-lazy-loading that image is the one part of this "performance" package that could measurably hurt perceived performance on the exact loop the game is built around. The package's risk assessment ("Low — ... sanitizer change is one function") doesn't account for this because it treats the sanitizer edit as mechanically trivial without checking what the images actually look like in the running app. I also flagged two smaller correctness gaps (trend-window cache staleness a small community would notice; a hand-hashed font-preload path that goes stale on next deploy) that a competent engineer should catch during implementation, not reasons to reject the package.

Net: the core P0 is real, verified, low-risk, and worth shipping this weekend as claimed — but ship it with these three corrections folded in, not as literally spec'd.

## Judge B (amend)
1. Fix the `_headers` syntax. The package's own "changes" text writes the rule as one line: `/assets/* Cache-Control: public, max-age=31536000, immutable`. Cloudflare Pages' `_headers` format requires the path pattern alone on its line with header directives indented below it:
   ```
   /assets/*
     Cache-Control: public, max-age=31536000, immutable
   ```
   Implemented literally as specified, this either fails to parse or is ignored, silently no-op'ing the exact P0 the package exists to fix. (Verified live: `curl -sI https://vwikirace.pages.dev/assets/index-BqFtIVtz.js` currently returns `cache-control: public, max-age=0, must-revalidate` with no `immutable` — the underlying bug is real, so this needs to actually land correctly.)

2. Add a `<Suspense>` boundary for the lazy `AdminDailies`, and update the test it will break. Confirmed via grep: there is zero `Suspense` usage anywhere in `src/` today. `AppShell.tsx` currently statically imports `AdminDailies` (line 2) and renders it directly (~line 152) inside a synchronous early-return. `React.lazy` requires an ancestor `<Suspense>` — omitting one (as the package does) is invalid and will throw/warn at runtime. Concretely, `src/App.test.tsx` (~lines 63-134) already navigates to `/admin/dailies` as an authorized admin and asserts on rendered admin content and fetch calls without awaiting a lazily-resolved chunk — this test will break unless the package (a) wraps `<AdminDailies>` in `<Suspense fallback=...>` and (b) updates that test to await the resolved chunk (e.g. `findBy*` queries). Neither is in the package's `files` list.

3. Drop or scope down trend-window caching — "finished trend windows" is not actually closed data. `Boards.tsx` has two independent effects: the board fetch (lines 235-252, `today`/`yesterday` via `getChallengeBoard`, keyed on `activeChallenge.id`) and the trends fetch (lines 254-269, `7d`/`30d`/`lifetime` via `getBoardsTrends`). `yesterday` is genuinely frozen once the day passes — caching it is safe. But 7d/30d/lifetime trends roll forward continuously and change the instant the *current session's own* race completes (finish today's daily → flip to Boards → your run should be in the rolling windows). A session Map cache on these, with no invalidation hook anywhere in `changes` or `acceptance`, will serve stale pre-completion trend data for the rest of the session. The one named acceptance test ("Bouncing Today→Yesterday→Today issues no repeat network call for the closed board") only exercises the safe half. Recommend: cache `yesterday`'s board only this cycle, or add an explicit cache-bust on run completion plus a named regression test for the staleness case.

4. Name the actual tests, not just curl/manual acceptance steps. The codebase has near-universal colocated test coverage, but `wikipediaSanitizer.ts` has zero test file today (grepped — none references it in any `*.test.*`), despite being the HTML sanitizer for untrusted Wikipedia content. Add `src/services/wikipediaSanitizer.test.ts` asserting `sanitizeAttributes` now unconditionally sets `loading="lazy" decoding="async"` on `<img>` regardless of source markup. Add a call-count assertion to the existing `src/modes/Boards.test.tsx` for the yesterday-cache-hit behavior (mock `apiClient.getChallengeBoard` call count across a today→yesterday→today bounce). The `curl -sI` / DevTools-network acceptance checks are fine as a post-deploy smoke step but shouldn't be the only verification for behavior that CI runs on every PR.

5. The font-preload sub-item isn't directly implementable as scoped — either narrow it or cut it. `@fontsource/luckiest-guy` and `@fontsource/fredoka` each ship 10+ per-subset/per-weight files (confirmed: hebrew/latin/latin-ext × multiple weights in `node_modules/@fontsource/*/files/`), imported as CSS in `main.tsx` and Vite-processed with content-hashed output filenames in `dist/assets/` that aren't knowable until build time. There is no single stable "the woff2" to hardcode a `<link rel=preload href=...>` for in a static `index.html`, and the package's `files` list has no manifest-reading build step or Vite plugin to resolve the hash. Either (a) pick exactly one file per font (e.g. latin/400 weight — a product call on language support, not just an engineering edit) and add a small postbuild script that reads Vite's manifest and injects the resolved preload href, or (b) drop the font-preload sub-item from this cycle and keep only the preconnect hints (which are static, safe, and correctly scoped — `en.wikipedia.org` + the hardcoded single-environment worker origin from `.env.production` are both fine as static edits).

Effort/risk: with amendments 2-5 folded in, this is still buildable this weekend, but call it Low-Medium risk rather than Low — the Suspense/test-breakage and trend-caching staleness issues are real correctness risks, not textbook boilerplate. The eager→lazy image-loading edit itself (sanitizer, ~1 function) and the corrected `_headers` two-line syntax are the clean, low-risk parts of this package.

### Judge B reasoning
Verified every file/line the package cites against the actual repo and, where possible, against the live site.

Confirmed real and correctly scoped: the P0 caching bug is genuine (curl against production shows `cache-control: public, max-age=0, must-revalidate` on a hashed JS asset, no `immutable`); this is Cloudflare Pages (`pages_build_output_dir = "./dist"`, no `assetsDir` override) so a `public/_headers` file with an `/assets/*` glob is mechanically the right mechanism and Vite's default `assets/` output makes the glob correct. The `sanitizeAttributes` img-loading change is a genuine one-function edit as described (verified `sanitizeTokenAttribute` today only whitelists pre-existing values, never adds missing ones). The `AdminDailies` gate (`isAdminDailiesRoute()`) is a clean natural split point for `React.lazy`. No conflict found with any round-1 shipped package (grepped the round-1 README/PKGs — none touch caching, headers, or AdminDailies loading strategy).

But the package is not implementable exactly as written in four concrete, checkable ways: (1) the `_headers` rule syntax shown is invalid for Cloudflare Pages' actual two-line format and would silently fail to fix the very bug it targets; (2) converting to `React.lazy` with zero `Suspense` anywhere in the codebase will break an existing, currently-passing admin-route test in `App.test.tsx` unless a Suspense boundary and test update are added — neither is scoped; (3) the acceptance criteria mischaracterize rolling trend windows as "closed" data safe to cache for a whole session, when they demonstrably change on the current user's own race completion, creating a real staleness bug with no invalidation path specified or tested; (4) the package leans on manual/curl verification for the two behavior-changing code edits in a codebase with near-total automated test coverage, and doesn't name the tests that should exist. A fifth issue narrows rather than blocks: the font-preload ask has no stable target file to preload given `@fontsource`'s per-subset/per-weight hashed output, so it needs either a build-step addition or trimming from scope.

None of this contradicts the diagnosis or the overall direction — the P0 is real, the P1/P2/P3 performance wins are reasonable targets, and the shape of the fix (headers file, sanitizer tweak, code-split admin, scoped cache, preconnect) is right. This is an amend, not a drop: land it with the syntax fixed, a Suspense boundary + test update added, trend-window caching removed or given explicit invalidation, named unit tests added for the two behavior changes, and the font-preload sub-item either narrowed to one concrete file via a postbuild step or cut from this pass.
