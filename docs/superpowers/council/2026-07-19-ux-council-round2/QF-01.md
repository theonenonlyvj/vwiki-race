# QF-01 [P0/S] Start gate defaults to Guest, not Create-account

## Files
- src/App.tsx:912 (openAuthPrompt fallback)
- src/App.tsx:1535-1541 (identity-copy paragraph)
- src/App.test.tsx:396-419
- src/App.test.tsx:731-762

## Changes
Flip the unconditional fallback in openAuthPrompt from setAuthMode("create") to setAuthMode("guest") — the guest tab already handles both cases (first-timer gets one name field; returning ghost gets one-tap 'Continue as guest'). Branch the identity-copy paragraph on authMode so the Guest tab reads 'Pick a name and go — claim it later' instead of 'Create a VGames account before the timer starts.' Update the two locked tests that currently encode the bug as spec. Leave explicit preferredMode callers (nominate-for-daily, You/Results claim CTAs) untouched.

## Acceptance
- Fresh visitor tapping Start race lands on the Guest tab: one name field + Race, no password fields visible
- Returning ghost gets the 1-click 'Playing as: X / Continue as guest' path with zero typing
- Guest tab copy no longer instructs account creation; Create/Log-in tabs keep the cross-device pitch
- Claim/nominate flows still open on Create/Log-in as before; full test suite green

## Risk
Low — the guest path is fully built and tested; only the default changes. Main cost is updating two copy-locked tests.

## Judge A (amend)
Cut this package down to ONLY the identity-copy branching fix (App.tsx:1535-1541), and drop the headline change — flipping openAuthPrompt's unconditional fallback from setAuthMode("create") to setAuthMode("guest") (App.tsx:912) — from this weekend's quickfix batch. Concretely:

1. KEEP, as P2/QUICKFIX: branch the identity-copy paragraph on `authMode` (not just `isGhost`) so switching to the Guest tab shows guest-appropriate copy instead of "Create a VGames account before the timer starts." This is real, uncontested, and cheap regardless of what the default tab is.

2. DROP/PARK, do not quickfix: the default-fallback flip (App.tsx:912) that sends every "Start race" tap to the Guest tab. I rendered and cropped the cited ratified mockup (council-shots/mockup-onboarding-v2.png, "STEP 2B — SAVE YOUR RUN") and it explicitly shows **"Create New" pre-selected with a teal focus ring** as the default tab in the identity sheet — for a brand-new first-time visitor, no less — with Guest positioned as a one-tap-away alternative via the footnote "Guest = instant, claim it later." The shipped code (and the two tests the package calls "locked-in bugs," esp. App.test.tsx:396 titled "defaults the start gate to a VGames Create account flow") is a faithful implementation of that ratified mockup, not an oversight. This is exactly the shape of thing round 1's own README already has a category for ("Deliberately dropped" #3: Race mode's sticky header — "The shipped layout matches the ratified mockup... this is a design-change proposal, not a drift bug. Needs an owner decision before anyone touches it"). Flipping a mocked, ratified, already-shipped default without a design conversation is re-litigation, which the owner explicitly asked this round not to do.

3. IF the owner separately wants to revisit the default, split out the one sub-case the mockup is silent on: a RETURNING ghost (identitySession.status === "ghost") lands on Create by default on every single Start, costing 2 taps (switch to Guest, then submit) to keep playing under a name they already have — every day, forever. The mockup only covers a first-time visitor, so this specific case isn't settled by ratified reference and is worth a narrow, explicit owner question ("should a returning ghost default to Guest while a brand-new visitor still defaults to Create?") rather than folding it into the same code change as the fresh-visitor case.

4. Scoping correction regardless of verdict: the package's stated risk ("only cost is updating two copy-locked tests") is incomplete. I found 5 more currently-green tests (App.test.tsx ~421, 447, 470, 499, 527) that interact with the create-account username field immediately after clicking "Start race" with no explicit tab click — i.e., they also implicitly depend on Create being the default. Any future default-flip would need one extra `click(getByRole("button", {name: /create account/i}))` line in each. Still S effort, but "files" should list this full set, not just the two cited blocks.

### Judge A reasoning
I verified the code, both cited tests, and the ratified mockup before judging.

Code (App.tsx:901-918, 858-869): confirmed — startSelectedChallenge's Start-race path calls openAuthPrompt with no preferredMode whenever there's no identity OR the identity is a ghost, which falls into the else branch and unconditionally sets authMode to "create." App.test.tsx:396-419 and :731-762 exist exactly as described and pass today, encoding this as current behavior. The identity-copy paragraph (App.tsx:1535-1541) does branch on ghost-status but never on authMode, so switching tabs manually leaves stale "create an account" copy — that half of the finding (the P2) is a genuine, verified bug independent of any default-tab debate.

The P0 half is where the package goes wrong. Its narrative frames the Create-account default as an unintended fallback bug ("the guest path is fully built and tested; only the default changes"). But I rendered the ratified onboarding mockup this project explicitly ships against (council-shots/mockup-onboarding-v2.png) and cropped the exact "SAVE YOUR RUN" identity sheet: "Create New" is the visually pre-selected tab (teal focus ring) for a brand-new first-time visitor, with Guest offered as a fast alternate path via a footnote, not as the default. The shipped code matches this mockup precisely. That means the P0 "fix" is actually a proposal to reverse a decision that was designed, mocked, and implemented as intended — squarely inside the "don't re-litigate decisions that shipped as designed" instruction for this round, and directly analogous to round 1's own "Deliberately dropped" precedent for the Race-mode sticky header, where the council explicitly declined to touch shipped-matches-mockup behavior without an owner conversation.

Net: this isn't a clean keep (the headline change is unjustified and risks exactly the re-litigation churn the owner said he's tired of) and it isn't a clean drop either (there's a real, safe, mockup-consistent copy bug buried inside, and a legitimate open question about returning-ghost friction that the mockup doesn't settle). Amend to ship only the copy-branching fix now, and route the default-tab question to the owner rather than auto-flipping it.
