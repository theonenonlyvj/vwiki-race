## Changes
Add 'color 140ms ease, background-color 140ms ease' to the base button transition list (one line, fixes every tab/nav/link hover sitewide). Add a shared modal entrance: backdrop opacity 0→1 (~140ms) + panel translateY(8px)/opacity (~180ms) on mount — exit stays a hard unmount this pass. Add a result-reveal keyframe (opacity+10px rise, ~260ms) on .race-results so the 'YOU REACHED IT' moment arrives instead of materializing. Add ~100-120ms opacity fades on the Boards tabpanel (keyed by segment), the identity form (keyed by authMode), and the trend drilldown. Add matching 'animation: none' lines to the existing prefers-reduced-motion block for every new keyframe.

## Acceptance
- All three dialogs (identity, End Run, how-to) fade/rise in; no hard pop-in anywhere in the modal system
- Mode-nav/tab/link color changes ease like their box-shadow already does
- Results has a visible entrance beat; segment and auth-form swaps read as deliberate
- prefers-reduced-motion renders everything instantly with zero animation

## Risk
Low — CSS-only plus one mount class; reduced-motion overrides specified up front.

The strong parts of this package are genuinely owner-aligned: the win moment ("YOU REACHED IT") currently materializing with zero entrance is a real miss on the app's one payoff beat in a daily-ritual game, yet it's buried as one bullet among five inside a P2 rather than called out as the headline item. ### Judge B reasoning
Verified every cited line range against the actual files and the plan is largely sound: styles.css:66-69 is exactly the base `button{}` transition list, :178-268 exactly spans modal-backdrop/identity-dialog/end-run-dialog/teaching-gate-dialog, :1234-1240 is the `.race-mode, .race-results` shared layout rule, Boards.tsx:435-439/465-499 is exactly the tabpanel div and the drilldown `<ol>`, and App.tsx:1589-1684 is exactly the three authMode-branched `<form>` blocks (which already fully mount/unmount on switch, so no `key` prop is actually needed there despite the "keyed by authMode" phrasing — harmless, not a bug). RaceResults.tsx:152-153 confirms `.race-results` is a genuine fresh-mount component per run, so a plain CSS `animation:` will fire correctly with no JS change. The Boards tabpanel genuinely needs the `key={segment}` the plan implies, since it's one persistent div whose children swap — verified no ref/focus/scroll logic is scoped to that div, so remounting it is safe, and Boards.test.tsx re-queries the DOM fresh each assertion rather than holding stale node references, so it won't break.

Three real gaps, all fixable without changing the package's shape:

1. Overstated blast radius. "one line, fixes every tab/nav/link hover sitewide" is false for actual `<a>` elements: `.site-footer a` (styles.css:807-820) has zero `transition` declared today and is a real `<a>`, not a `<button>` — the base-button edit won't touch it, yet the acceptance bar explicitly says "Mode-nav/tab/link color changes ease" and names "link." PreRacePreview.tsx's attribution `<a>` has the same gap. Scope must either add `.site-footer a { transition: color 140ms ease, border-bottom-color 140ms ease; }` or the acceptance criterion needs to drop "link."

2. Selector spillover risk. `.race-mode, .race-results` (styles.css:1233-1240) is a *combined* selector for two mutually-exclusive RaceFlow branches (confirmed at RaceFlow.tsx:178-206). The plan's own line citation points at that combined block, but adding `animation:` there in place would also animate `.race-mode` (the live race view) on every race start, which isn't what "the win moment" was scoped for. Needs a new standalone `.race-results { animation: ...; }` rule instead of editing the cited block in place — worth stating explicitly so whoever implements it doesn't just tack animation onto line 1234.

3. Reduced-motion approach has a live precedent failure. The plan's mitigation is "add matching 'animation: none' lines... for every new keyframe" (5-6 new keyframes: backdrop fade, panel rise, result-reveal, tabpanel fade, drilldown pop). Checked the codebase's one existing `@keyframes` today (`article-loading`, styles.css:1465-1468, `infinite alternate`) and confirmed it is **not** covered by the current `prefers-reduced-motion` block (styles.css:3045-3057), which only neutralizes `.article-navigation-pending::after` — i.e. this exact enumerate-per-keyframe pattern has already produced a real miss in this codebase (an infinite-loop animation that ignores reduced-motion today). Recommend amending to a blanket `*, *::before, *::after { animation-duration: 0.01ms !important; animation-iteration-count: 1 !important; }` inside the existing block instead of per-keyframe overrides — same effort, structurally can't be missed, and incidentally fixes the pre-existing article-loading gap.

4. No test requirements named. Everything else here is pure CSS, fine to ship without new tests, but `key={segment}` on the Boards tabpanel is a real DOM-structural/behavioral change, not just styling. The package should explicitly name "Boards.test.tsx and App.test.tsx pass unmodified" as the regression gate for that one change, since it's the only place this touches more than CSS.

None of this changes effort (still M) or risk tier (still Low-ish) — it's an amend, not a drop.
