- src/modes/challenges/Browse.tsx:237,270-272 (badge) + visibleChallenges (~257)
- src/modes/challenges/ChallengeDetail.tsx:84

## Changes
Route Browse and ChallengeDetail badges through 'challenge.dailyFeature ? dailyFlavorBadgeText(challenge.dailyFeature) : dailyBadgeLabel(...)' so dailies read 'Hard · Daily #4' like Home/Boards/Preview (non-dailies keep the date/Today label). Route the RaceMode article kicker and RaceResults embedded-panel kicker through the same dailyNumberLabel(...) ?? challenge.label pattern shared.tsx:120 already proved out. Exclude pinnedDaily?.id from Browse's All Challenges list so today's daily stops appearing twice.

## Acceptance
- The same daily shows one vocabulary (flavor + Daily #N) on Home, Boards, Preview, Browse pinned card, Browse catalog card, Challenge Detail, in-race kicker, Results kicker, and share text
- A friend clicking a 'Daily #4' share link lands on a page that says Daily #4
- Pinned daily no longer duplicates into All Challenges; non-daily challenges' labels unchanged

## Risk
Low — both helpers exist, are tested, and are used elsewhere; expect a handful of copy-locked test string updates.

## Judge B (amend)
1. FIX THE FRAMING BUG on the pinned card (Browse.tsx:237): the package's literal call `dailyFlavorBadgeText(challenge.dailyFeature)` omits the `framing` arg, which defaults to `"today"` (signature: `dailyFlavorBadgeText(dailyFeature, framing: "today"|"yesterday" = "today")`, src/domain/dailyEditorial.ts:113). Home.tsx:213 and Boards.tsx:361 both thread `hero.kind`/`todayShowsYesterdayFraming` into that second arg specifically so the badge reads "Yesterday's daily · Flavor · Daily #N" during a pre-drop/countdown state. Browse already has the equivalent signal (`heroSelection.kind`, checked at Browse.tsx:156) but the package's snippet doesn't pass it through. As literally written, this ships the exact live state named in this council's evidence (07-19 daily not yet dropped, Home showing yesterday's daily) with Home's hero reading "Yesterday's daily · Recognizable · Daily #4" and Browse's pinned card reading bare "Recognizable · Daily #4" — contradicting the package's own acceptance criterion #1 ("same vocabulary... on Home, Boards, Preview, Browse pinned card"). Required: `dailyFlavorBadgeText(pinnedDaily.dailyFeature, heroSelection.kind === "yesterday-daily" ? "yesterday" : "today")`.

2. NARROW the ChallengeDetail.tsx:84 / Browse.tsx:270-272 (catalog-row) swap or name its cost explicitly. Both sites currently use `dailyBadgeLabel`, which shows "Today" for the actual current daily and an actual calendar date ("Daily 7/15") for any other daily — the only place in the app that distinguishes "this specific daily IS today's" from "this is some other day's daily," since neither Detail nor the catalog list has Home/Boards' accompanying heading/aria-label to carry that signal separately. `dailyFlavorBadgeText` has no generalized "N days ago" framing (only a today/yesterday binary built for Home/Boards' single-active-daily context) — applying it uniformly to Detail/catalog rows that can show ANY daily, arbitrarily old, makes every dailyFeature-carrying daily render identically ("Recognizable · Daily #4") regardless of whether it's today's or three weeks stale. That may be an intentional "Daily #N is the identity, Wordle-style" call (consistent with PKG-07's own framing), but the package doesn't say so, and it silently breaks an existing, semantically-loaded regression test: src/App.test.tsx:2579, `within(detail).getByText("Today")`, added specifically as "the proxy for 'the merge preserved dailyFeature metadata'" (see comment at App.test.tsx:2576-2577). That test isn't in the package's files list, and the acceptance criteria don't call it out — the Risk section's "expect a handful of copy-locked test string updates" undersells this one, since it's checking a real invariant (dailyFeature metadata survived a merge), not decorative copy. Before shipping: either get explicit owner sign-off that Detail/catalog no longer need to signal "is this today's actual daily" (then update App.test.tsx:2579 to assert on the new flavor+number text AND independently verify the merge still preserves dailyFeature, e.g. via `flavor`/`dailyNumber` values rather than deleting the check), or keep `dailyBadgeLabel` for Detail/catalog rows and confine the vocabulary unification to the pinned card + kicker + share text (which is what the "duplicate pinned daily" and "Challenge #1 kicker" findings actually require).

3. Sequencing/imports (trivial, note for the implementer): Browse.tsx currently computes `visibleChallenges` (line ~146-149) before `pinnedDaily` (line ~156-158) — excluding `pinnedDaily?.id` from the catalog list requires either reordering these two blocks or otherwise threading the id through. Also name the new imports: `dailyFlavorBadgeText` and `dailyNumberLabel` both live in `src/domain/dailyEditorial.ts`, not `src/domain/challengeSelection.ts` (which only exports `dailyBadgeLabel`) — Browse.tsx/ChallengeDetail.tsx need a new import line alongside the existing `dailyBadgeLabel` import; RaceMode.tsx/RaceResults.tsx need a first-time import of `dailyNumberLabel`. None of this blocks the fix, but "right files" should list `src/domain/dailyEditorial.ts` (read, not edited) and `src/App.test.tsx` (edited) alongside the four files already named.

Everything else in the package checks out on file-read verification: the RaceMode.tsx:129/184 and RaceResults.tsx:351 kicker citations are exact and the fix (swap in `dailyNumberLabel(challenge.dailyFeature?.dailyNumber) ?? challenge.label` at the existing string-prop computation site) is low-risk and precedented by shared.tsx:120's `composeShareText` — no existing test asserts a literal "Challenge #N" kicker on a dailyFeature-carrying challenge, so that half of the package is implementable as written. The duplicate-pinned-daily bug is real and verified: `visibleChallenges` (Browse.tsx) is derived from the full `challenges` prop with no exclusion, so today's pinned daily currently appears a second time in "All challenges" underneath it.
